# AI 策略预期合理性审查

> **审查我们预设的期望是否合理、充分且必要**
> 
> 创建时间: 2026年1月2日

---

## 审查目标

重新评估7个测试场景中的预期策略，确保：
1. **逻辑自洽**：预期策略符合业务逻辑
2. **充分性**：决策依据足够支撑判断
3. **必要性**：不包含过度约束
4. **稳健性**：在真实场景中可行

---

## Part 1: 场景1（低产慢消费）预期审查

### 原始预期
```
- cooldownMinutes >= 120（避免推荐堆积）
- analysisIntervalMinutes >= 10（低产出不需频繁分析）
- scoreThreshold >= 7.0（质量优先）
```

### 数据回顾
```json
{
  "dailyNewArticles": 10,
  "rawPoolSize": 25,
  "candidatePoolSize": 8,
  "dailyReadCount": 2.5,
  "likeRate": 60.0,
  "dismissRate": 10.0
}
```

### 合理性分析

#### ✅ cooldownMinutes >= 120 的合理性

**论证**：
- 用户日均消费2.5篇 → 消费间隔 = 1440 / 2.5 = 576分钟（9.6小时/篇）
- 如果 cooldown = 60分钟，推荐池补充速度 = 24次/天
- 但用户只消费2.5篇/天，补充过快会导致堆积
- **cooldown >= 120分钟**意味着最多12次补充/天，更接近消费速度

**结论**：✅ **合理**，符合"推荐频率应匹配消费速度"的原则

---

#### ⚠️ analysisIntervalMinutes >= 10 的合理性

**原始论证**：
- 低产出（10篇/天）不需频繁分析

**反驳**：
- rawPoolSize = 25，dailyNewArticles = 10
- 积压天数 = 25 / 10 = **2.5天**
- 这是否算"积压"？需要清理吗？

**重新评估**：
```
情况A：认为2.5天积压需要清理
→ analysisInterval 应较短（5-15分钟）快速处理

情况B：认为2.5天积压是正常缓冲
→ analysisInterval 可较长（30-60分钟）慢慢消化
```

**关键问题**：rawPool的"合理大小"是多少天储备？

**我的判断**：
- 对于低产源（10篇/天），2.5天积压 = 25篇
- 候选池8篇 = 3.2天消费储备（健康）
- 推荐池2篇 = 0.8天消费储备（略低）
- **rawPool 25篇不算严重积压**，可以中速处理

**修正预期**：
- ~~analysisIntervalMinutes >= 10~~
- **analysisIntervalMinutes in [15, 45]**（中速处理，既不激进也不拖延）

**结论**：⚠️ **原预期过于保守**，应允许更灵活的范围

---

#### ✅ scoreThreshold >= 7.0 的合理性

**论证**：
- likeRate = 60%，dismissRate = 10%
- 质量指数 = (60×2 + 90) / 3 = 70
- 用户满意度高，应保持质量标准
- threshold = 7.0-7.5 合理

**结论**：✅ **合理**

---

### 场景1修正后的预期

**原预期**：
```
- analysisIntervalMinutes >= 10
- cooldownMinutes >= 120
- scoreThreshold >= 7.0
```

**修正预期**：
```
- analysisIntervalMinutes in [15, 45]（中速处理，不激进不拖延）
- cooldownMinutes in [90, 180]（宽松范围，匹配慢消费）
- scoreThreshold in [7.0, 7.5]（保持质量）
- dailyLimit in [6, 12]（略高于消费量，预留缓冲）
```

**修正理由**：
1. 2.5天积压不算严重，不需≤10分钟激进处理
2. 慢消费应给更宽松的冷却范围（90-180而非仅120+）
3. 增加dailyLimit约束，防止AI过度推荐

## Part 2: 场景2（高产快消费）预期审查

### 原始预期
```
- analysisIntervalMinutes <= 5
- batchSize >= 15
- cooldownMinutes <= 60
- scoreThreshold in [7.5, 8.0]
- dailyLimit >= 20
- candidatePool.targetSize >= 50
```

### 数据回顾
```json
{
  "dailyNewArticles": 150,
  "rawPoolSize": 200,
  "candidatePoolSize": 15,
  "dailyReadCount": 12.0,
  "likeRate": 45.0,
  "dismissRate": 25.0
}
```

### 合理性分析

#### ✅ analysisIntervalMinutes <= 5 的合理性

**论证**：
- 积压天数 = 200 / 150 = 1.33天
- 但每天新增150篇，如果不快速处理，积压会继续增长
- 候选池仅15篇 = 1.25天消费储备（不足）
- 需要快速补充候选池

**数学验证**：
```
当前生产速度：150篇/天 = 6.25篇/小时
当前消费速度：12篇/天 = 0.5篇/小时
当前分析速度（假设interval=5分钟，batch=15）：
  = (60/5) × 15 = 180篇/小时（理论最大值）
  = 考虑通过率40% → 72篇/小时进入候选池

如果interval=15分钟：
  = (60/15) × 15 = 60篇/小时
  = 通过率40% → 24篇/小时进入候选池
  仍远大于消费速度0.5篇/小时
```

**反思**：
- 候选池需要的补充速度 = 0.5篇/小时（消费速度）
- 即使 interval=30分钟，也能满足
- **为什么需要 <= 5分钟？**

**重新评估**：
1. **清理积压视角**：200篇积压需要快速清理 → interval <= 5合理
2. **供需平衡视角**：只需匹配消费速度 → interval 可放宽至15-30分钟

**我的判断**：
- 候选池15篇偏低（< 2天储备），需要优先补充
- rawPool 200篇积压不算严重（1.33天），但持续新增150篇/天有压力
- **interval <= 5分钟偏激进**，但考虑到高产出压力，可以接受

**结论**：✅ **基本合理**，但可以考虑放宽至 **<= 10分钟**

---

#### ⚠️ scoreThreshold in [7.5, 8.0] 的争议

**原始论证**：
- 质量指数 = (45×2 + 75) / 3 = 55（一般）
- 应提高阈值至7.5-8.0保证质量

**AI的判断**：
- threshold = 6.8（初测）或 7.8（增强规则后）

**争议点**：
- likeRate = 45%，dismissRate = 25%
- 这是否意味着"质量差，需提高阈值"？
- 还是"用户需求大，可适度放宽"？

**数学验证**：
```
如果 threshold = 8.0：
- 假设通过率降至20%
- 每天分析150篇 → 30篇进入候选池
- 用户消费12篇/天 → 候选池增长 18篇/天
- 7天后候选池 = 15 + 18×7 = 141篇（过剩）

如果 threshold = 7.0：
- 假设通过率40%
- 每天分析150篇 → 60篇进入候选池
- 用户消费12篇/天 → 候选池增长 48篇/天
- 候选池会快速过剩
```

**关键洞察**：
- **高产出场景下，threshold不能太低，否则候选池过剩**
- **但也不能太高，否则用户无内容可看**
- likeRate = 45%可能不是因为推荐质量差，而是用户品味挑剔
- dismissRate = 25%在快消费场景下可以接受（75%完成率）

**修正判断**：
- threshold = 7.5-8.0 **可能过高**
- threshold = 7.0-7.5 更平衡
- 或者用动态策略：初期7.0，候选池充足后提升至7.5

**结论**：⚠️ **原预期可能过于严格**，建议 **[7.0, 7.5]** 或 **[7.2, 7.8]**

### 场景2修正后的预期

**原预期**：
```
- analysisIntervalMinutes <= 5
- scoreThreshold in [7.5, 8.0]
- cooldownMinutes <= 60
```

**修正预期**：
```
- analysisIntervalMinutes in [5, 10]（快速但不极端）
- scoreThreshold in [7.0, 7.5]（平衡质量与供应）
- cooldownMinutes in [30, 60]（匹配快消费）
- batchSize in [12, 18]（高效批量）
- candidatePool.targetSize in [40, 60]（3-5天储备）
```

**修正理由**：
1. interval <= 5过于激进，<= 10更稳健
2. threshold 7.5-8.0可能导致供应不足，7.0-7.5更平衡
3. 增加目标范围而非单一下限，更灵活
