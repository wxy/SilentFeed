## 1. 产品概述

### 1.1 产品名称
**SilentFeed** (Feed + AI + Muter)

**命名含义**：
- **Feed**: RSS 订阅源
- **AI**: 人工智能推荐
- **Muter**: 静音器，过滤噪音

### 1.2 一句话描述
一个 AI 驱动的浏览器扩展，自动学习你的浏览习惯，从 RSS 订阅中筛选出真正感兴趣的内容，以克制的方式提醒你。

### 1.3 产品标语
- **中文**: RSS，由 AI 重新定义 / 静默你的信息流
- **英文**: The AI that reads RSS for you - intelligently muting the noise.

### 1.4 产品理念

**核心思想**：不是让用户主动管理 RSS，而是 AI 作为"信息守门员"

1. **静默学习**：在后台自动分析用户浏览行为，不打扰用户
2. **智能过滤**：只推送真正感兴趣的内容，其余静默处理
3. **隐私优先**：所有分析默认在本地进行，用户完全掌控数据
4. **克制设计**：不追求"上瘾"，只在有价值时才出现
5. **渐进成长**：通过 1000 页面冷启动，逐步了解用户

### 1.5 核心价值主张

| 维度 | 传统 RSS 阅读器 | SilentFeed |
|------|----------------|-------------|
| **信息过滤** | 手动分类和过滤 | AI 自动筛选 |
| **个性化** | 基于用户设置 | 基于实际行为 |
| **交互模式** | 主动打开查看 | 被动等待提醒 |
| **学习成本** | 需要配置管理 | 零配置自动学习 |
| **信息负担** | 未读累积焦虑 | 只看重要内容 |

## 2. 核心问题与解决方案

### 2.1 用户痛点

#### 痛点 1: 信息过载
**场景**: 订阅了 50+ RSS 源，每天新增 200+ 条内容
**问题**: 
- 没时间全部看完
- 一键"全部已读"后有遗漏重要内容的焦虑
- 逐条浏览太耗时

**我们的解决方案**: 
AI 自动筛选出最相关的 3-5 条，其余静默处理

#### 痛点 2: 推荐不准确
**场景**: 依赖人工设置标签和分类
**问题**:
- 兴趣会变化，分类规则过时
- 无法表达复杂的兴趣模式
- 冷启动困难（新用户不知道如何设置）

**我们的解决方案**:
基于真实浏览行为自动学习，动态适应兴趣变化

#### 痛点 3: 工具成为负担
**场景**: RSS 阅读器需要频繁打开管理
**问题**:
- 需要主动打开查看
- 订阅管理繁琐
- 成为另一个"待办清单"

**我们的解决方案**:
被动式守门员，只在必要时提醒，其余时间完全静默

### 2.2 为什么现在是时机

1. **AI 成熟**: LLM 能理解内容语义，推荐准确度显著提升
2. **隐私意识**: 用户更关注数据隐私，本地 AI 成为可能
3. **信息焦虑**: 注意力经济时代，克制的产品更受欢迎
4. **技术基础**: Chrome Extension MV3 + Web AI API 成熟

## 3. 目标用户

### 3.1 种子用户（优先级 1）

**人群**: 重度 RSS 用户 + 效率工具爱好者

**特征**:
- 已订阅 30+ RSS 源
- 使用 Feedly/Inoreader 等工具
- 有信息过载困扰
- 愿意尝试新工具
- 对隐私敏感

**规模**: 10 万 - 50 万（全球）

**获取渠道**:
- Hacker News
- Reddit (r/rss, r/selfhosted)
- ProductHunt
- 少数派

### 3.2 早期采用者（优先级 2）

**人群**: 知识工作者、研究者、内容创作者

**特征**:
- 需要持续获取行业信息
- 时间宝贵，注重效率
- 可能不了解 RSS，但有信息获取需求
- 愿意为工具付费

**规模**: 100 万 - 500 万

**获取渠道**:
- 博客和播客推广
- 与知名创作者合作
- LinkedIn 等职业社交平台

### 3.3 主流用户（优先级 3）

**人群**: 普通互联网用户

**特征**:
- 不了解 RSS 概念
- 依赖算法推荐（今日头条、Twitter）
- 对现有推荐质量不满意
- 希望有更好的信息获取方式

**规模**: 数千万

**策略**: 淡化"RSS"概念，强调"智能信息管家"

### 6.4 RSS 源管理策略

**用户主动订阅 vs 自动发现的不同处理**：

```

  ├─ 采取保守策略，尽量不干预
  ├─ 长期无更新（30 天）→ 提示用户
  ├─ 推荐但从不阅读（20 次 0 互动）→ 建议静音
  └─ 长期无推荐（90 天）→ 询问是否取消

  ├─ 采取激进策略，快速验证价值
  ├─ 初始 10 条内容无推荐 → 自动移除
  ├─ 推荐后无互动 → 降权
  └─ 多次推荐+高互动 → 转为"优质源"
```

## 7. 功能详细说明

### 7.1 浏览历史收集

**目标**：静默收集用户浏览行为，构建兴趣画像

**实现细节**：
- 监听页面访问（通过 content script）
- 提取页面特征：
  - 标题、URL、域名
  - 正文内容（前 5000 字）
  - 停留时间（超过 30 秒才记录）
  - 用户交互（复制、收藏、分享等）
- 过滤无效页面：
  - 域名黑名单（邮件、银行、内网等）
  - 非内容页面（登录页、404 等）
  - 停留时间过短（< 30 秒）
- 隐私保护：
  - 只存储提取的特征，不存储完整 HTML
  - 敏感域名自动排除
  - 用户可配置排除规则

**有效页面定义**：
```typescript
function isValidPage(page: PageData): boolean {
  return (
    page.readingTime > 30 &&           // 停留超过 30 秒
    page.wordCount > 300 &&            // 至少 300 字
    !isExcludedDomain(page.url) &&     // 不在黑名单
    page.contentType === 'article'     // 是文章内容
  )
}
```

### 7.2 用户画像构建

**目标**：基于浏览行为自动构建用户兴趣画像

**核心算法**：
1. **TF-IDF 文本分析**：提取关键词
2. **主题分类**：使用预定义主题模型（技术、科学、商业等）
3. **时间衰减**：近期浏览权重更高
4. **行为加权**：深度交互（复制、收藏）权重更高

**画像数据结构**：
```typescript
interface UserProfile {
  topics: {                    // 主题概率分布
    'technology': 0.35,
    'science': 0.28,
    'design': 0.18,
    'business': 0.12,
    'other': 0.07
  },
  keywords: string[],          // Top 50 关键词
  domains: string[],           // 常访问域名
  behaviorScore: number,       // 行为评分（0-1）
  temporal: {                  // 时间模式
    recent_30d_weight: 0.6,
    last_90d_weight: 0.3,
    older_weight: 0.1
  }
}
```

### 7.3 RSS 自动发现

**目标**：检测用户访问的网站是否提供 RSS 源

**实现方式**：
```typescript
// 检测页面 <link> 标签
<link rel="alternate" type="application/rss+xml" href="/feed.xml">
<link rel="alternate" type="application/atom+xml" href="/atom.xml">

// 常见 RSS URL 模式
/feed
/rss
/feed.xml
/atom.xml
/index.xml
```

**自动订阅策略**：
- 检测到 RSS 源 → 提示用户是否订阅
- 用户访问频率高的网站 → 自动添加到候选列表
- 试用期：抓取前 10 条内容测试推荐效果
- 效果好 → 正式订阅，效果差 → 自动移除

### 7.4 AI 推荐引擎

**目标**：从大量 RSS 内容中筛选出用户最感兴趣的 3-5 条

**三层推荐架构**：

```
Layer 1: 本地预筛选（TF-IDF）
  输入: 200 条新 RSS 内容
  处理: 计算与用户画像的相似度
  输出: 30 条候选（Top 15%）
  优点: 快速、隐私、免费
  
Layer 2: 用户 API 评分（可选）
  输入: 30 条候选
  处理: LLM 深度理解内容和用户兴趣
  输出: 每条内容的匹配度分数（0-1）
  优点: 准确度高（85-95%）
  成本: 极低（< $0.01/天）
  
Layer 3: Chrome AI 增强（可选）
  输入: AI 评分结果
  处理: 本地 AI 二次验证
  输出: 最终推荐列表
  优点: 完全免费、隐私
```

**推荐质量对比**：
- 纯 TF-IDF：准确率 ~50%，召回率 ~40%
- TF-IDF + LLM：准确率 ~85%，召回率 ~75%

### 7.5 智能通知机制

**目标**：在合适的时机提醒用户，避免打扰

**通知策略**：

```typescript
// 通知评分算法
function calculateNotificationScore(
  recommendation: Recommendation,
  userState: UserState
): number {
  return (
    recommendation.importance * 0.35 +      // 内容重要性
    calculateTimeliness() * 0.25 +          // 时效性
    userState.availability * 0.20 +         // 用户当前状态
    historicalEngagement * 0.15 +           // 历史互动率
    dailyQuotaRemaining * 0.05              // 今日剩余配额
  )
}

// 通知时机判断
if (score > 0.7) {
  showPopup()          // 弹窗通知
} else if (score > 0.4) {
  showBadge()          // 图标徽章
} else {
  keepSilent()         // 保持静默，用户主动查看
}
```

**用户状态检测**：
- 正在写作/看视频 → 不打扰（score = 0）
- 空闲 1-5 分钟 → 好时机（score = 0.9）
- 用户活跃时段 → 可以提醒（score = 0.7）

**每日配额**：
- 默认最多 5 次弹窗通知
- 图标徽章无限制
- 用户可自定义配额

### 7.6 反馈循环

**目标**：根据用户反馈持续优化推荐

**反馈类型**：
1. **明确反馈**：
   - 点击阅读 → 正反馈（+1）
   - 标记"不喜欢" → 负反馈（-1）
   - 稍后阅读 → 中性（0.5）
   
2. **隐式反馈**：
   - 阅读时长 > 2 分钟 → 深度兴趣（+2）
   - 复制内容 / 收藏 → 非常感兴趣（+3）
   - 打开后立即关闭 → 不感兴趣（-0.5）

**学习机制**：
```typescript
// 更新画像
async function updateProfile(feedback: Feedback) {
  if (feedback.type === 'like') {
    increaseWeight(item.keywords, 1.2)
    increaseWeight(item.source, 1.1)
  } else if (feedback.type === 'dislike') {
    decreaseWeight(item.keywords, 0.8)
    decreaseWeight(item.topics, 0.9)
  }
  
  // 重新计算推荐
  await refreshRecommendations()
}
```

- 🌱 初始化中（绿色种子 + 进度环）
- 😴 静默中（灰色，表示无推荐）
- 🔔 有推荐（橙色 + 数字徽章）
- ⚡ 紧急推荐（红色，重要内容）

### 8.2 游戏化元素

**成长阶段可视化**：
```
🌱 探索者 (0-250 页)
  └─ 解锁: 基础推荐

🌿 学习者 (251-600 页)
  └─ 解锁: 智能时机判断

🌳 成长者 (601-1000 页)
  └─ 解锁: 高级分析

🌲 大师 (1000+ 页)
  └─ 解锁: 完整能力
```

**成就系统**：
- 📚 阅读新手：完成 100 页分析
- 🎯 精准推荐：10 次推荐全部阅读
- 🌟 信息大师：累计节省 50 小时
- 🦄 稀有发现：找到 3 个小众优质源

## 10. 非功能性需求

### 10.1 性能要求
- 页面分析耗时 < 100ms
- 推荐引擎响应 < 2s
- RSS 抓取不阻塞浏览
- 内存占用 < 100MB

### 10.2 隐私和安全
- **默认本地处理**: 所有分析在用户设备完成
- **可选云端 AI**: 只上传"概率云"数据（主题分布），不上传原始浏览记录
- **用户 API**: 数据直接发给用户选择的 AI 服务商，不经过我们的服务器
- **域名黑名单**: 自动排除敏感网站（银行、邮件、医疗等）
- **数据加密**: API Key 加密存储
- **随时删除**: 用户可一键删除所有历史数据

### 10.3 可用性
- 安装后自动工作，无需配置
- AI 可选，不强制
- 倒计数降低用户期待
- 清晰的数据使用说明

### 10.4 可扩展性
- 模块化架构，易于添加新功能
- 支持多种 AI 提供商
- 预留云端服务接口
- 支持插件/主题扩展（V2）

## 13. 风险与挑战

### 13.1 隐私信任风险

**风险**: 用户不信任扩展会收集浏览历史

**缓解措施**:
- 开源核心代码
- 清晰的隐私政策
- 提供完全本地化选项
- 用户可查看所有收集的数据
- 明确说明数据用途和存储位置

### 13.2 推荐准确度风险

**风险**: 推荐不准确导致用户流失

**缓解措施**:
- 冷启动倒计数降低期待
- 提供反馈机制快速学习
- 支持手动调整偏好（V2）
- A/B 测试不同推荐算法
- 持续优化模型

### 13.3 AI 成本风险

**风险**: 使用云端 AI 成本过高

**缓解措施**:
- MVP 阶段使用用户 API（成本转嫁）
- 本地预筛选减少 AI 调用量（200 → 30）
- 选择便宜的模型（DeepSeek, gpt-4o-mini）
- 批量处理降低请求次数
- 引入超售模式（V3）

### 13.4 技术挑战

**挑战**: Chrome Extension MV3 限制

**解决方案**:
- Service Worker 生命周期管理
- 使用 IndexedDB 持久化数据
- 优化后台任务调度
- 参考 Plasmo 最佳实践

### 13.5 竞争风险

**风险**: Feedly 等传统 RSS 阅读器增加 AI 功能

**差异化**:
- 我们是被动式，他们是主动式
- 我们基于浏览行为，他们基于声明兴趣
- 我们更克制，不追求"上瘾"
- 我们完全开源透明
